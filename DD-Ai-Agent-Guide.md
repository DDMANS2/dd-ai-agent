# DD Ai 에이전트 — 종합 개발 지침서

> 최종 업데이트: 2026-03-22
> 이 문서는 DD Ai 에이전트 프로젝트의 전체 스펙, 아키텍처, 프롬프트, 기술 스택을 포함합니다.

## 1. 프로젝트 개요

### 1-1. 한 줄 설명
여러 AI 에이전트가 각자 역할(진행자, 전략가, 개발자 등)을 맡아 사용자의 주제에 대해 실제 회의처럼 토론하고, 자동으로 결론과 실행 가이드를 도출하는 웹 애플리케이션.

### 1-2. 핵심 기능 목록
| 기능 | 설명 |
|------|------|
| 에이전트 11명 | 회의 진행자, 전략가, 크리에이터, 리스크 분석가, 실행 매니저, 데이터분석가, 마케터, 재무분석가, 개발자, 디자인 분석가, 영업 전문가 |
| 에이전트 초대/퇴장 | 클릭으로 회의 참여자 관리 (기본 5명 ON) |
| 수동 모드 | 사용자가 토론/결론 타이밍 직접 컨트롤 |
| 자동 수렴 모드 | 3라운드 자동 토론 → 결론 자동 도출 |
| 회의 템플릿 | 브레인스토밍, 의사결정, 회고/리뷰, 전략 수립 |
| 웹 검색 모드 | 에이전트가 실시간 데이터 검색 (Anthropic/Perplexity) |
| 결론 리포트 | 합의/다수/미해결 + 실행 로드맵 + 도구/예산 + 리스크/KPI |
| 개발 지침 자동 생성 | 개발 키워드 감지 시 바이브 코딩 가이드 자동 포함 |
| 다중 LLM 지원 | 내장 Claude, Cerebras, Groq, OpenAI, Anthropic, Perplexity |
| 음성 입출력 | STT (음성→텍스트), TTS (텍스트→음성) |
| 반응형 UI | 데스크톱/모바일 자동 대응 |
| 도움말 | 6탭 FAQ 형식 |
| 회의록 | 실시간 기록 + 텍스트 파일 내보내기 |

## 2. 아키텍처

### 2-1. 현재 구조 (단일 JSX 아티팩트)
```
meeting-room-v3.jsx (단일 파일, ~2200줄)
├── AGENTS[] — 에이전트 11명 정의
├── MEETING_TEMPLATES[] — 회의 템플릿 4종
├── LLM_PROVIDERS[] — LLM 제공자 6종
├── HelpModal — 도움말 모달
├── AgentAvatar — 에이전트 아바타 컴포넌트
├── ChatBubble — 채팅 버블 컴포넌트
├── ConclusionReport — 결론 리포트 (5탭)
├── Section — 섹션 렌더링 헬퍼
├── ConvergenceProgress — 자동 수렴 진행 바
├── useIsMobile — 모바일 감지 훅
└── MeetingRoomV3 — 메인 컴포넌트
    ├── callLLM() — 에이전트 대화 API 호출
    ├── callLLMRaw() — 범용 LLM 호출
    ├── callForConclusion() — 결론 전용 호출 (안정성 우선)
    ├── safeParseJSON() — 잘린 JSON 복구
    ├── generateConclusion() — 2단계 결론 생성 + 개발 지침
    ├── runOneRound() — 에이전트 1라운드 실행
    ├── runAutoConvergence() — 자동 수렴 3라운드
    ├── handleSend() — 메시지 전송
    ├── toggleVoice() — STT 음성 입력
    ├── speakText() — TTS 음성 출력
    └── exportReport() — 리포트 내보내기
```

#### 라운드 2: 토론 & 반박
```
userMessage:
이전 라운드의 다른 에이전트 의견을 검토하고, 구체적으로 이름을 
언급하며 동의/반박/보완해주세요. 네 입장이 바뀌었다면 왜 바뀌었는지도 설명해.

extraSystem:
이것은 토론 라운드야. 반드시 다른 에이전트의 이름을 직접 언급하며 
동의하거나 반박해. 예: '전략가의 의견에 동의하지만, 리스크 분석가가

```

#### 라운드 3: 수렴 & 합의
```
userMessage:
2라운드의 토론을 바탕으로, 합의할 수 있는 부분은 합의하고, 
양보할 부분은 양보해주세요. 최종 입장을 정리해주세요.

extraSystem:
이것은 수렴 라운드야. 2라운드 토론을 통해 생각이 바뀐 부분이 있으면 
솔직히 밝혀. '처음엔 반대했지만, [에이전트]의 의견을 듣고 조건부 
찬성으로 바꿉니다'처럼. 합의 가능한 공통분모를 적극 찾되, 
진짜 양보 못하는 부분은 명확히 해.
```

## 4. 결론 생성 프롬프트

### 4-1. 1단계: 결론 + 실행 로드맵
```
시스템 프롬프트:
회의 결론 도출 전문가. 아래 JSON 형식으로만 답변. JSON 외 텍스트 없이. 
완전한 JSON 필수.

{
  "title": "주제 요약",
  "consensus": ["전원 동의 사항"],
  "majority": ["다수 동의 사항"],
  "unresolved": ["미해결 쟁점"],
  "finalConclusion": "최종 결론 3-5문장",
  "executionRoadmap": [{
    "phase": "단계명",
    "period": "기간",
    "goal": "목표",
    "tasks": [{
      "task": "할일",
      "howTo": "구체적 방법",
      "tools": ["도구"],
      "cost": "비용",
      "duration": "소요시간",
      "difficulty": "쉬움/보통/어려움",
      "tip": "팁"
    }]
  }]
}

규칙: 3단계, 각 tasks 2개, howTo는 구체적으로
```

### 4-2. 2단계: 도구/예산/리스크
```
시스템 프롬프트:
실행 정보를 JSON으로.

{
  "techStack": [{"category":"분야", "recommended":"도구", "free":"무료", "reason":"이유"}],
  "budget": {"minimum":"최소", "recommended":"권장", "breakdown":["내역"]},
  "risks": [{"risk":"리스크", "prevention":"대응"}],
  "kpi": ["목표"],
  "resources": ["자료"]
}
```

### 4-3. 3단계: 개발 지침 (자동 감지 시)
```
감지 키워드: 개발, 시스템, 앱, 웹사이트, 플랫폼, 서비스, API, 
자동화, 봇, 대시보드, MVP, 프로토타입, SaaS, 솔루션, 기능 구현, 소프트웨어

시스템 프롬프트:
시니어 풀스택 개발자이자 바이브 코딩 전문가. JSON으로 작성.

{
  "devGuide": {

```

## 7. 핵심 로직 상세

### 7-1. 대화 히스토리 병합
```
목적: Anthropic API는 연속된 같은 role 메시지를 허용하지 않음

1. 최근 14개 메시지 추출
2. user 메시지는 그대로, assistant 메시지는 "[에이전트명]: 내용" 형식
3. 연속된 같은 role 메시지를 \n으로 병합
4. 마지막이 user가 아니면 user 메시지 추가
```

### 7-2. JSON 안전 파싱 (safeParseJSON)
```
LLM 응답이 잘릴 수 있으므로 복구 로직 필요:

1차: 정상 JSON.parse
2차: 열린 괄호({, [) 카운트 후 닫기
3차: 마지막 쉼표 기준으로 잘라내고 괄호 닫기
실패: null 반환
```

### 7-3. 결론 생성 안정화
```
문제: 3라운드 × 11에이전트 = 33메시지 → 너무 길어서 서버 오류

해결:
1. 대화 내용 3000자 제한 (앞 1500 + 뒤 1500)
2. 결론 전용 API 함수 (웹검색 없이 가벼운 호출)
3. 3단계 재시도: 정상 → 짧은 버전 → 최소 형태
4. 결론/도구/개발지침을 각각 별도 API 호출
```
